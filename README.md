# Silvers Fun

Silvers Fun is a Flutter mobile app that helps seniors connect with people, discover shared interests, and join or create small meetups.

The app is designed for adults 65+ who want a warm, simple, and low-pressure way to meet friends, companions, and activity partners.

## What the app does

Silvers Fun helps users:

- Create a personal profile with photos, bio, age, and interests

- Discover other members with shared interests

- Like profiles and form mutual connections

- Chat after a mutual connection

- Browse, create, join, and leave meetups

- Manage profile photos and choose a main profile photo

- Re-pick interest tags after onboarding

- Switch between English and Korean in account settings

## Why it exists

Many seniors want companionship, conversation, and shared activities, but existing social or dating apps can feel too fast, too confusing, or too focused on younger users.

Silvers Fun is built around:

- Companionship, not swipe pressure

- Shared interests and activities

- Simple profile controls

- Mutual connections before chat

- Large, readable, senior-friendly UI

- Korean and English accessibility

## Releasing to Play Store internal testing

```sh
scripts/build_release_aab.sh
# AAB lands at build/app/outputs/bundle/release/app-release.aab
# Then: Play Console → Internal testing → Create new release → upload that AAB.
```

Google Sign-In works because `com.google.gms.google-services` reads
`android/app/google-services.json` at build time and emits a
`default_web_client_id` string resource that `google_sign_in` picks up
automatically. For a Play-signed AAB to authenticate, the **App Signing
SHA-1** from Play Console (Test and release → App integrity → App signing
key) must be added to the Firebase Android app, then `flutterfire configure`
re-run to refresh `google-services.json`.

## Handoff / Project ownership transfer

When transferring this project to a different developer/publisher who will own
the Android app under a different `applicationId`, do the following **before**
handing off the repo:

1. Run the rename script and pick the new package name (e.g. `com.acmecorp.someapp`):

   ```sh
   python3 scripts/rename_android_package.py
   ```

   It updates `android/app/build.gradle.kts` (namespace + applicationId), the
   three `AndroidManifest.xml` files, `android/app/google-services.json`, and
   moves/rewrites the Kotlin source tree under `android/app/src/main/kotlin/`.

   It deliberately does **not** touch `lib/firebase_options.dart`,
   `pubspec.yaml`, or any iOS files — those are either regenerated by tooling
   or left as a separate concern.

2. Commit the rename in a single commit and push to whatever git remote the
   new owner will clone from.

After cloning, the new owner must:

1. Create a new Firebase project in their own account at
   https://console.firebase.google.com

2. Add an Android app to that Firebase project with the new package name from
   the rename script.

3. Add their debug SHA-1 to that Firebase Android app (get it via):

   ```sh
   cd android && ./gradlew signingReport
   ```

4. Run `flutterfire configure` — this rewrites
   `android/app/google-services.json` and `lib/firebase_options.dart` with
   their Firebase project's values.

5. Run `flutter pub get`.

6. Generate their own upload keystore:

   ```sh
   keytool -genkey -v -keystore android/upload-keystore.jks \
       -keyalg RSA -keysize 2048 -validity 10000 -alias upload
   ```

7. Create `android/key.properties` (gitignored) with their keystore details:

   ```
   storeFile=upload-keystore.jks
   storePassword=<their store password>
   keyAlias=upload
   keyPassword=<their key password>
   ```

8. Create the app in their own Play Console developer account and enroll in
   Play App Signing.

9. Build the release AAB:

   ```sh
   ./scripts/build_release_aab.sh
   ```

10. Upload the AAB to internal testing in their Play Console.

11. Once enrolled in Play App Signing, copy the **App Signing SHA-1** from
    Play Console and add it to their Firebase Android app, then re-run
    `flutterfire configure` and rebuild before promoting to production.
