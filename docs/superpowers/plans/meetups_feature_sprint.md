# Meetups Feature Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a simple meetup scheduling feature where any signed-in user can create a meetup, browse upcoming meetups created by others, and join/leave a meetup. Organizers can cancel their own meetup. Fully bilingual (English + Korean) from day one.

**Architecture:** New `meetups` top-level Firestore collection. One document per meetup, with `signedUpUids: List<String>` carrying the attendee list (excluding the organizer). One repository (`MeetupsRepository`) injected with `FirebaseFirestore`, three Riverpod providers (`meetupsRepositoryProvider`, `upcomingMeetupsProvider`, `meetupByIdProvider`), three screens (`UpcomingMeetupsScreen`, `CreateMeetupScreen`, `MeetupDetailScreen`), wired into the existing `ShellRoute` as a new bottom-nav tab. Tests use `FakeFirebaseFirestore` (already a dev-dep). Localization uses the existing `gen-l10n` ARB pipeline. Firestore rules are extended locally only — **no `firebase deploy` step in this sprint.**

**Tech Stack:** Flutter 3.11.5+ · `cloud_firestore` 5.4 · `flutter_riverpod` 2.5 · `go_router` 14.2 · `flutter_localizations` (EN + KO) · `fake_cloud_firestore` (tests) · existing `Btn`, `ChipTag`, `PhotoWidget`, `AppColors`, `AppTheme`, `showToast` — no new third-party deps.

## Sprint goal

Ship a v1 meetup feature for Silvers Fun adults 65+:

- Any signed-in user can create a meetup with title, description, start date/time, location text, and an optional max-attendees cap.
- Any signed-in user can see a list of upcoming, non-canceled meetups, sorted by start time.
- Any signed-in user can join/leave a meetup (other than their own), respecting capacity if set.
- The organizer is shown a "You're hosting" badge and a "Cancel meetup" action; canceling soft-removes the meetup from the upcoming list.
- The current user's join/host status is visible at a glance on each list card and on the detail screen.
- All user-visible strings exist in both English and Korean ARB files; the device locale drives which one renders.
- Firestore rules forbid impersonation (only the organizer can edit/cancel their own meetup; only the user themselves can add/remove their own uid from `signedUpUids`).
- The existing 65/65 tests stay green; new repository tests cover the create/join/leave/cancel paths against `FakeFirebaseFirestore`.

## Non-goals

- No GPS, maps, or place autocomplete. Location is plain text.
- No payments, RSVPs with notes, "maybe" responses, or waitlists.
- No recurring events, calendar sync, or .ics export.
- No push or email notifications.
- No moderator dashboard, reporting, or blocking.
- No per-meetup chat thread or comments.
- No host approval workflow (joining is instant up to capacity).
- No major redesign of existing tabs.
- No `firebase deploy` — rules are updated in `firestore.rules` locally only and the user will deploy later.
- No package rename (`name: silver_fun` in `pubspec.yaml` stays).

## Recommended UI placement

**Add a new "Meetups" bottom-nav tab** as the third destination, between "Liked you" and "Chats" (final order: Discover · Liked you · Meetups · Chats · You — five tabs total).

**Why this over the alternatives:**

- **vs. "accessible from You/Discover":** meetups are a primary, frequent action (browse → join → host), not a settings sub-page. Burying it one tap deeper would hurt discovery for the 65+ audience that is the whole reason for this feature.
- **vs. "replace/rename an existing tab":** every existing tab is load-bearing. Replacing "Liked you" would orphan the reverse-like UX and break the existing badge contract; replacing "Discover" is a non-starter; merging "You" into Settings creates a long modal. Adding a fifth tab keeps every prior flow working unchanged.
- **Acceptable senior-UX cost:** Material 3 `NavigationBar` supports up to five destinations cleanly. The tab order above keeps Discover/Liked-you/Chats/You in the same relative positions; Meetups just inserts itself in the middle. Existing icon/label conventions (outline → filled, l10n label) are reused.

**Future-proof exit:** if user research shows five tabs feel crowded, a follow-up sprint can collapse "Liked you" into "You" and free a slot. No schema change required for that pivot.

## Firestore schema

**New top-level collection:** `meetups`

| Field | Type | Required? | Notes |
|---|---|---|---|
| `id` (doc id) | `String` (auto) | yes | Auto-generated by Firestore on `add()`. |
| `organizerUid` | `String` | yes | Must equal `request.auth.uid` at create time. Cannot be changed after. |
| `title` | `String` | yes | Trimmed. Client-side cap 80 chars. |
| `description` | `String` | yes (may be empty) | Trimmed. Client-side cap 500 chars. |
| `startsAt` | `Timestamp` | yes | Must be in the future at create time (client-side check). |
| `location` | `String` | yes | Trimmed. Client-side cap 120 chars. Plain text only. |
| `maxAttendees` | `int?` | optional | `null` means no cap. Must be ≥ 1 if set. Capped at 200 client-side as a sanity bound. |
| `signedUpUids` | `List<String>` | yes | Always present; created as empty list `[]`. Organizer is **not** auto-included. Length is the displayed attendee count. |
| `canceled` | `bool` | yes | Always present; created as `false`. Set to `true` by organizer to soft-cancel. Canceled meetups are filtered out of the upcoming list. |
| `createdAt` | `Timestamp` | yes (server-set) | Written via `FieldValue.serverTimestamp()` at create time. |

**Why `signedUpUids` as an array on the parent doc (instead of a subcollection):**

- Single-doc atomic update via `FieldValue.arrayUnion([uid])` / `arrayRemove([uid])`.
- Attendee count is just `signedUpUids.length` — no extra read.
- `hasJoined(uid)` is `signedUpUids.contains(uid)` — no extra read.
- Firestore array membership is checkable in security rules (`hasAny`, `hasAll`, `.size()`).
- For senior-companionship meetups, expected size is well under the 1MB doc soft-cap (≤ ~50 attendees realistically); array scaling is not a concern at v1 volume.

**Indexes:** the upcoming-meetups query `where('canceled', isEqualTo: false).where('startsAt', isGreaterThan: now).orderBy('startsAt')` requires a composite index on `(canceled ASC, startsAt ASC)`. Firestore will offer to auto-create it on the first uncached query in dev — note this in the QA checklist.

## Security rules plan

Append a new block to `firestore.rules` **locally** (no deploy in this sprint). The rules forbid:

1. Anyone other than the organizer creating a doc with someone else's `organizerUid`.
2. Anyone other than the organizer changing the meetup's title/description/start/location/maxAttendees/canceled.
3. Anyone other than the organizer deleting the meetup.
4. Anyone adding/removing a uid in `signedUpUids` other than their own (i.e., no signing other people up or kicking other people out).
5. Anyone joining a full meetup (capacity check enforced server-side).
6. Anyone modifying immutable fields (`organizerUid`, `createdAt`) after creation.

**Final rules block to insert** (between the `chats` block and the `match /{document=**}` deny-all at the bottom):

```
// Meetups — anyone signed-in can read; only organizer creates/edits/cancels;
// non-organizers can only join/leave (toggle their own uid in signedUpUids).
match /meetups/{meetupId} {
  allow read: if isSignedIn();

  allow create: if isSignedIn()
    && request.resource.data.organizerUid == request.auth.uid
    && request.resource.data.title is string
    && request.resource.data.title.size() > 0
    && request.resource.data.location is string
    && request.resource.data.startsAt is timestamp
    && request.resource.data.signedUpUids is list
    && request.resource.data.signedUpUids.size() == 0
    && request.resource.data.canceled == false;

  allow update: if isSignedIn() && (
    isMeetupOrganizerEdit() || isMeetupJoinOrLeave()
  );

  allow delete: if isSignedIn()
    && resource.data.organizerUid == request.auth.uid;
}

function isMeetupOrganizerEdit() {
  return resource.data.organizerUid == request.auth.uid
    && request.resource.data.organizerUid == resource.data.organizerUid
    && request.resource.data.signedUpUids == resource.data.signedUpUids
    && request.resource.data.createdAt == resource.data.createdAt;
}

function isMeetupJoinOrLeave() {
  let me = request.auth.uid;
  let curr = resource.data.signedUpUids;
  let next = request.resource.data.signedUpUids;
  let onlyChange = request.resource.data.diff(resource.data).affectedKeys().hasOnly(['signedUpUids']);
  let cap = resource.data.get('maxAttendees', 0);
  let withinCapacity = cap == 0 || next.size() <= cap;
  let isJoin = !curr.hasAny([me])
    && next.hasAll(curr)
    && next.hasAny([me])
    && next.size() == curr.size() + 1
    && withinCapacity;
  let isLeave = curr.hasAny([me])
    && curr.hasAll(next)
    && !next.hasAny([me])
    && next.size() == curr.size() - 1;
  return onlyChange && (isJoin || isLeave);
}
```

`maxAttendees == 0` is treated as "no cap" in the rule (matches `null` in the doc since `.get('maxAttendees', 0)` returns the default if absent). Client always writes either an int ≥ 1 or omits the field entirely.

**Deploy step is intentionally absent.** When the user is ready to deploy, they will run:

```bash

```

This is **not** part of this plan's automation.

```dart
import 'package:cloud_firestore/cloud_firestore.dart';

class Meetup {
  final String id;
  final String organizerUid;
  final String title;
  final String description;
  final DateTime startsAt;
  final String location;
  final int? maxAttendees;
  final List<String> signedUpUids;
  final bool canceled;
  final DateTime? createdAt;

  const Meetup({
    required this.id,
    required this.organizerUid,
    required this.title,
    required this.description,
    required this.startsAt,
    required this.location,
    this.maxAttendees,
    this.signedUpUids = const [],
    this.canceled = false,
    this.createdAt,
  });

  factory Meetup.fromFirestore(DocumentSnapshot<Map<String, dynamic>> doc) {
    final data = doc.data() ?? const <String, dynamic>{};
    return Meetup(
      id: doc.id,
      organizerUid: (data['organizerUid'] as String?) ?? '',
      title: (data['title'] as String?) ?? '',
      description: (data['description'] as String?) ?? '',
      startsAt: (data['startsAt'] as Timestamp?)?.toDate() ??
          DateTime.fromMillisecondsSinceEpoch(0),
      location: (data['location'] as String?) ?? '',
      maxAttendees: (data['maxAttendees'] as num?)?.toInt(),
      signedUpUids: ((data['signedUpUids'] as List?) ?? const [])
          .whereType<String>()
          .toList(),
      canceled: (data['canceled'] as bool?) ?? false,
      createdAt: (data['createdAt'] as Timestamp?)?.toDate(),
    );
  }

  bool isOrganizer(String uid) => organizerUid == uid;
  bool hasJoined(String uid) => signedUpUids.contains(uid);
  int get attendeeCount => signedUpUids.length;
  bool get isFull =>
      maxAttendees != null && signedUpUids.length >= maxAttendees!;
}
```

import 'package:cloud_firestore/cloud_firestore.dart';

import '../models/meetup.dart';

class MeetupsRepository {
  final FirebaseFirestore _db;

  MeetupsRepository(this._db);

  CollectionReference<Map<String, dynamic>> get _meetups =>
      _db.collection('meetups');

  Future<String> createMeetup({
    required String organizerUid,
    required String title,
    required String description,
    required DateTime startsAt,
    required String location,
    int? maxAttendees,
  }) async {
    final ref = await _meetups.add({
      'organizerUid': organizerUid,
      'title': title,
      'description': description,
      'startsAt': Timestamp.fromDate(startsAt),
      'location': location,
      if (maxAttendees != null) 'maxAttendees': maxAttendees,
      'signedUpUids': const <String>[],
      'canceled': false,
      'createdAt': FieldValue.serverTimestamp(),
    });
    return ref.id;
  }

  Stream<List<Meetup>> watchUpcoming({DateTime? now}) {
    final cutoff = Timestamp.fromDate(now ?? DateTime.now());
    return _meetups
        .where('canceled', isEqualTo: false)
        .where('startsAt', isGreaterThan: cutoff)
        .orderBy('startsAt')
        .snapshots()
        .map((snap) => snap.docs.map(Meetup.fromFirestore).toList());
  }
}

}

function isChatParticipant(chatId) {

function isMeetupOrganizerEdit() {
  return resource.data.organizerUid == request.auth.uid
    && request.resource.data.organizerUid == resource.data.organizerUid
    && request.resource.data.signedUpUids == resource.data.signedUpUids
    && request.resource.data.createdAt == resource.data.createdAt;
}

function isMeetupJoinOrLeave() {
  let me = request.auth.uid;
  let curr = resource.data.signedUpUids;
  let next = request.resource.data.signedUpUids;
  let onlyChange = request.resource.data.diff(resource.data).affectedKeys().hasOnly(['signedUpUids']);
  let cap = resource.data.get('maxAttendees', 0);
  let withinCapacity = cap == 0 || next.size() <= cap;
  let isJoin = !curr.hasAny([me])
    && next.hasAll(curr)
    && next.hasAny([me])
    && next.size() == curr.size() + 1
    && withinCapacity;
  let isLeave = curr.hasAny([me])
    && curr.hasAll(next)
    && !next.hasAny([me])
    && next.size() == curr.size() - 1;
  return onlyChange && (isJoin || isLeave);
}

## Acceptance criteria

A reviewer should confirm each of these in order:

1. **Builds clean.** `flutter analyze` reports no issues. `flutter test` reports all tests passing (≥ 79).
2. **Bottom nav.** App shows five tabs in order: Discover · Liked you · Meetups · Chats · You. The Meetups tab uses an event icon (outline → filled).
3. **Localization.** When the device locale is English, every meetups screen reads in English. When the device locale is `ko`, every meetups screen reads in Korean (incl. validation errors, toasts, empty states).
4. **Create flow.** Tapping `+` on the Meetups list opens a form. Validation rejects empty title, missing date/time, past date/time, empty location, and `maxAttendees` < 1. Submitting valid input writes a doc to `meetups/{id}` with the documented schema, shows toast "Meetup created", and returns to the list.
5. **List flow.** The list shows future, non-canceled meetups ordered by start time. Each card shows title, formatted date/time, location, attendee count, and either a "Hosting" or "Joined" pill if applicable. The list updates live when another user creates or cancels a meetup.
6. **Detail flow — non-organizer.** Tapping a card opens the detail. If the user has not joined and the meetup is not full, a "Join" button is shown; tapping it adds their uid, shows toast "You joined the meetup", and the count updates live. If they have joined, "Leave" replaces "Join" and behaves symmetrically. If the meetup is full and they haven't joined, the button label is "Meetup full" and disabled.
7. **Detail flow — organizer.** Detail shows "You're hosting" and a "Cancel meetup" button. Tapping it sets `canceled = true`, shows toast "Meetup canceled", and pops back to the list (where the meetup no longer appears).
8. **Security rules (manual sanity, after the user deploys).** A non-organizer cannot edit anything except adding/removing their own uid. A non-organizer cannot delete the meetup. Capacity is enforced at the rule layer (joining a full meetup fails server-side, not just client-side).
9. **No regressions.** Existing flows (auth → onboarding → discover → like → chat → settings) work end-to-end on a fresh install. The "Liked you" badge and "Chats" unread badge still increment correctly (covered by the updated `MainShell` tests with the new index math).

## Risks / things to watch

1. **Composite Firestore index.** The first run of `watchUpcoming` will fail (or block) until Firestore creates the composite index `(canceled ASC, startsAt ASC)`. The `cloud_firestore` SDK surfaces a console URL to auto-create it. Document this in the QA notes; in dev, the index typically builds in <30s.
2. **Rules NOT deployed by this sprint.** The user explicitly chose to defer the deploy. Joins and creates will fail in production until they run `firebase deploy --only firestore:rules`. Surface this prominently in the PR description.
3. **`signedUpUids` array growth.** Capped at maxAttendees client-side, but the doc has a 1MB Firestore soft-cap. At ~30 chars per uid, that's well over 30k attendees — a non-issue at v1 scale. Revisit if we ever ship public meetups with thousands of slots.
4. **Capacity race.** Two users tapping "Join" simultaneously when only one slot remains: the rule's `next.size() <= cap` is checked at write-time, so the second writer's update fails server-side. Client UI may briefly show a stale "Join" button — acceptable for v1. If we get bug reports, switch to a transaction-based join with a re-read.
5. **Time zones.** Meetup `startsAt` is stored as a Firestore `Timestamp` (UTC). The display function in `_formatDateTime` uses local time via `DateTime` getters — i.e., the device's local time zone. This works for the senior single-region audience; if the app ever ships internationally with cross-timezone meetups, we'll need explicit zone handling.
6. **Date/time picker localization.** `showDatePicker` and `showTimePicker` use the active `MaterialApp` locale automatically. Korean device locale gets Korean pickers for free — verify in QA.
7. **Five-tab nav.** Material 3 `NavigationBar` supports five destinations but labels can wrap on narrow phones. If "Liked you" + "Meetups" labels overlap on small Androids, consider switching to `labelBehavior: NavigationDestinationLabelBehavior.onlyShowSelected` for that build.
8. **Internal symbol bleed.** Don't accidentally name new symbols using "match" prefixes (a leftover from the dating-app history). Use "meetup" everywhere — that's the user-facing word.
9. **Existing `MainShell` index math.** The badge logic uses hardcoded indices (1 = Liked you, 2 = Chats). After insertion, Chats becomes 3. The plan explicitly fixes this, but a future tab insertion will require revisiting again. Logged as a small risk; not worth a refactor in this sprint.
10. **No widget tests for new screens.** Repository + model coverage is good, but a regression in screen wiring (provider override, route mismatch, missing l10n key) only surfaces in the smoke run. If post-merge bugs appear here, add minimal `pumpWidget` tests as a follow-up.
